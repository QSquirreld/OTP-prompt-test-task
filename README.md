# LLM-оценка мотивации клиента на оплату задолженности

Проект посвящён построению и тестированию prompt-based решения для автоматической оценки блока **«Мотивация клиента на оплату»** в диалогах операторов отдела взыскания.

Задача: по тексту диалога и сроку просрочки определить, насколько оператор корректно мотивировал клиента на оплату задолженности с учётом категории клиента, согласия/несогласия на оплату, озвученных последствий, запрещённых формулировок и исключений.

## Используемый подход

В качестве baseline используется подход **один диалог → один LLM-запрос → JSON-оценка**.

На вход модели передаются:

- текст диалога;
- срок просрочки;
- правила определения категории клиента;
- правила оценки мотивации;
- списки последствий;
- исключения;
- требуемый JSON-формат ответа.

На выходе ожидается JSON вида:

```json
{
  "criterion_met": true,
  "decision": "выполнено",
  "client_category": "A",
  "client_status": "согласен оплатить",
  "exception_detected": false,
  "interrupted_dialog": false,
  "detected_consequences": [],
  "violations": [],
  "explanation": "Краткое объяснение результата."
}
```

## Основные выводы

Baseline-подход позволяет быстро получить рабочее решение и проверить применимость LLM к задаче речевой аналитики.

При этом в процессе тестирования были выявлены ограничения монолитного prompt'а:

- модель может путать согласие клиента с качественной мотивацией со стороны оператора;
- модель иногда придумывает последствия, которых не было в репликах оператора;
- модель может корректно объяснить проблему, но ошибиться в итоговом поле `criterion_met`;
- длинный prompt с большим количеством уточнений может перегружаться и ухудшать качество;
- часть проверок лучше выполнять детерминированно, а не отдавать LLM.

Главный инженерный вывод: один prompt подходит как baseline, но для production-версии лучше использовать гибридный пайплайн из правил и LLM-извлечения.

## Как интерпретировать результаты

Основная метрика в тестах — accuracy по ожидаемому полю `criterion_met`.

Также отдельно анализируются промежуточные признаки:

- `client_category`;
- `client_status`;
- `exception_detected`;
- `interrupted_dialog`;
- `detected_consequences`;
- `violations`.

Это важно, потому что итоговый ответ может быть случайно верным, но промежуточная логика модели — ошибочной.

## Ограничения текущего решения

Текущее решение является baseline и имеет несколько ограничений:

- зависит от качества LLM и стабильности API;
- чувствительно к формулировкам prompt'а;
- может ошибаться на синонимах и неполных репликах;
- не использует аудио-признаки, интонацию, паузы и перебивания;
- предполагает, что роли `Оператор` и `Клиент` уже корректно размечены;
- не гарантирует стабильное качество на реальных промышленных данных без дополнительной валидации.

## Возможные улучшения

Для production-версии рекомендуется перейти от одного монолитного prompt'а к гибридной архитектуре:

1. **Детерминированные правила**
  - определение категории клиента по сроку просрочки;
  - проверка отсутствия прощания;
  - часть проверок запрещённых формулировок.
2. **LLM extraction**
  - извлечение последствий из реплик оператора;
  - определение статуса клиента;
  - распознавание исключений;
  - обработка синонимов и сложных формулировок.
3. **Финальная агрегация**
  - сбор промежуточных результатов;
  - применение правил ТЗ;
  - формирование итогового JSON.
4. **Демо-интерфейс**
  - Streamlit или Flask-приложение;
  - поле для ввода диалога;
  - поле для срока просрочки;
  - вывод итоговой оценки, найденных последствий и нарушений.
5. **Сравнение моделей**
  - тестирование на GigaChat;
  - сравнение разных OpenAI-совместимых моделей;
  - анализ качества, скорости и стоимости.
